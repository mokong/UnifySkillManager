# Building A Unified Skill And Rule Manager For AI Coding Tools

Different AI coding tools have started to grow their own configuration ecosystems.

Codex has Skills. Claude has project and user instructions. Cursor has Rules. CodeBuddy has its own Skill layout. Other tools follow similar patterns, but the file locations, formats, naming conventions, and global/project scopes are different enough that managing them by hand quickly becomes messy.

UnifySkillManager is a small local Web app built to solve that problem: keep Skills and Rules in one Markdown-first library, then sync them to the tools that need them.

## The Problem

Most AI coding tools now support some version of reusable instructions:

- Skills for repeatable workflows
- Rules for coding conventions and behavior constraints
- Project-level instructions
- Global user preferences
- Tool-specific prompt fragments

The hard part is not writing the content. The hard part is remembering where every tool wants it.

One Skill might need to live in a folder structure like:

```txt
~/.codex/skills/my-skill/SKILL.md
```

Another tool might want:

```txt
~/.claude/skills/my-skill.md
```

Cursor rules may use:

```txt
.cursor/rules/my-rule.mdc
```

Once you want both global and project-level configuration, the mental overhead grows again.

## The Core Idea

UnifySkillManager treats Markdown as the source of truth.

Each Skill or Rule is just a `.md` file. Structured metadata lives in YAML frontmatter, while the actual instruction content remains normal Markdown.

```md
---
id: swiftui-review
type: skill
name: SwiftUI Review
description: Review SwiftUI code for maintainability and performance.
targets:
  - codex
  - cursor
scope:
  - global
  - project
enabled: true
version: 1.0.0
---

# SwiftUI Review

Use this skill when reviewing SwiftUI code.
```

This gives the project a few useful properties:

- The content is easy to read.
- The files are Git-friendly.
- The format can be edited outside the app.
- The Web UI can still parse and manage the metadata.
- Other import formats can be converted into Markdown later.

## What The Web UI Does

The local Web UI focuses on a few workflows.

First, it can import existing Markdown files. That is the primary path because many Skills and Rules already exist as Markdown somewhere on the machine.

Second, it can scan known tool directories and import deduplicated candidates. The scanner is conservative: it scans common Codex, Claude, Cursor, and CodeBuddy locations rather than crawling the whole computer.

Third, it lets you edit metadata and content in one place:

- ID
- Name
- Description
- Tags
- Target tools
- Global or project scope
- Markdown body

Finally, it can preview and sync generated files into target tool directories.

## Syncing Is More Than Copying

When you click sync, UnifySkillManager does a few things:

1. Saves the current metadata back to the source Markdown file.
2. Chooses the correct target path for each selected tool.
3. Renders tool-specific output.
4. Creates missing directories.
5. Backs up existing target files.
6. Writes the new file.
7. Reads the file back to verify it was written.

Every generated file includes a marker:

```md
<!-- Managed by UnifySkillManager. Manual edits may be overwritten. -->
```

That marker allows the UI to count managed files and makes it clear which files are generated by the manager.

## Tool Layouts

The current adapters target these layouts:

| Tool | Skill output | Rule output |
| --- | --- | --- |
| Codex | `skills/{id}/SKILL.md` | `rules/{id}.md` |
| Claude | `skills/{id}.md` | `rules/{id}.md` |
| Cursor | Rules-focused | `{id}.mdc` |
| CodeBuddy | `skills/{id}/SKILL.md` | `rules/{id}.md` |

This adapter-based design keeps the core model simple while letting each tool evolve independently.

## Why Local First

This project is intentionally local:

- No account system
- No cloud sync
- No external database
- No dependency-heavy stack

The source library is just files:

```txt
library/
  skills/
  rules/
backups/
config/
```

That makes it easy to inspect, back up, and version control.

## What Counts As Success

There are two levels of success.

The first is file-system success: UnifySkillManager wrote the generated file to the correct path, read it back, and confirmed the marker exists.

The second is tool-level success: the target tool actually loads and applies the Skill or Rule.

The app can verify the first one directly. The second depends on each tool's loading behavior. Some tools may need a window reload, workspace reload, or a new session before newly synced configuration is available.

That distinction matters. A manager should be honest about what it can verify.

## Next Steps

Useful future improvements include:

- Tool-specific schema validation
- Better Cursor `.mdc` frontmatter support
- Import conflict review
- Per-tool compatibility warnings
- A dedicated sync history page
- More adapters for Windsurf, Trae, Kiro, and Antigravity

## Closing Thought

AI coding tools are becoming programmable environments. As that happens, Skills and Rules become part of a developer's personal toolchain.

UnifySkillManager is a small attempt to make that toolchain portable, inspectable, and easier to reuse across tools.
